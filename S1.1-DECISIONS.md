## 1. Prefix algorithm (`<proj4>`)

**Decision:** `slugify(project).slice(0, 4)`, where `slugify` =
`lowercase → replace [^a-z0-9] with empty → collapse → take first 4 chars`.

If the slugified project name is **shorter than 4 characters**, pad on
the right with `0` (zero) until length 4. Example: `'q'` → `'q000'`.

The padding character is `0` (digit) so the prefix stays inside the
project-id charset (`a-z0-9`) and stays unambiguous w.r.t. existing
project names.

## 2. Prefix-collision resolution

**Decision:** **First-come-first-served deterministically at handle-mint time.**

`mintHandle` is the single place where a project's prefix is fixed. The
first time a canonical ID for project X is minted:

1. Compute the raw prefix `slugify(project).slice(0, 4)` (with the 0-pad
   from §1 above).
2. If that prefix is **not yet** in `counters`, use it as-is.
3. If it **is** already used (by a different project), suffix-digit
   walk: `<prefix-without-last-char><N>` where `N` starts at `2` and
   increments. So `tracker` colliding with `trafficGame` (both raw
   prefix `trac`/`traf`? no — `tracker` → `trac`, `trafficGame` → `traf`,
   no collision). A real collision would be e.g. `trafficX` clashing
   with `trafficY`: both raw prefix `traf` → second project gets
   `tra2`, then `tra3`, etc. The walk caps at 99 suffixes; beyond that
   we throw because the operator should pick a better project name.
4. Project→prefix binding is **persisted in `_aliases.json`** under
   `by_project[project] = prefix` so subsequent mints for the same
   project pick up the same suffix-walked prefix without re-running
   the walk.

**Why first-come:** picking the suffix at mint time is the only
choice that keeps the handle deterministic from local state alone
(no global registry of "what prefix would project X get"). The
operator's `forge enqueue` warns when a suffix walk fires — that's
the human's signal to rename the new project before it becomes
load-bearing.

The plan's `07b-init-ids.md` §"Open question 3" asks whether to
warn-then-mint or silently-mint. **Picked warn-then-mint** (Q3 had
this as the recommendation).

## 3. Ambiguous-name resolution semantics

The plan says (line 53): *"Resolution is **read-only against the
registry**; ambiguity prints all matches + exits 2."*

But `resolveInitiativeId` is a library function — it can't `exit(2)`
without coupling the resolver to the CLI. **Picked:** library returns
`null` for ambiguous and exposes the matches via a richer return:

```ts
type ResolveResult =
  | { kind: 'ok'; canonical: string; handle: string; name?: string }
  | { kind: 'ambiguous'; matches: string[] }
  | { kind: 'not-found' };

function resolveInitiativeId(input: string): ResolveResult;
```

The CLI wraps this and `exit(2)` on ambiguity, printing the matches.
Tests cover both shapes.

## 4. `_aliases.json` schema

Locked per C6/plan §"Storage + collision":

```json
{
  "version": 1,
  "by_handle":    { "traf#7":  "INIT-2026-05-19-trafficgame-backpressure-live" },
  "by_canonical": { "INIT-...": { "handle": "traf#7", "name": null } },
  "by_name":      { "backpressure-live": "INIT-..." },
  "by_project":   { "trafficgame": "traf" },
  "counters":     { "traf": 7, "bett": 2 }
}
```

Additions over the plan-stated schema:

- `by_name` — secondary index for the named-alias lookup path (`slugify-batch`
  → canonical). Without this, lookup is O(n) over `by_canonical`.
- `by_project` — pins the project → prefix mapping after the
  collision-walk so the prefix stays stable.

Both are derivable from `by_canonical` + `counters`, so the registry
self-heals on rebuild (the backfill script regenerates them from
scratch).

## 5. Atomic-write strategy

Per C17: `proper-lockfile` wraps the read → mutate → write sequence in
`mintHandle`. Reads (`loadAliases`) are **unlocked** per C17.

Lock acquisition uses default retries (`proper-lockfile` retries with
exponential backoff for ~10s by default). The lockfile lives at
`_queue/_aliases.json.lock`.

If the registry file doesn't exist yet, `lockfile` is told to consider
it as the lockable resource via the `realpath: false` option. On first
mint we create the parent dir, write an empty registry, then lock.

## 6. Parse-failure semantics

Per C16b spirit (idempotent replay over silent skip): a corrupt or
unparseable `_aliases.json` is treated as **empty registry**. The
backfill / mint flow then re-derives everything on the next write. This
is logged to stderr at warn level so the operator notices.

(A more conservative choice would be to refuse to proceed. Picked
"treat as empty" because the registry is purely a lookup cache — every
canonical ID is independently authoritative on disk under `_queue/`, so
worst-case we re-mint the same handles in the same order from the
backfill.)

## 10. Backfill of existing `_queue/`

The backfill script walks `_queue/{pending,in-flight,ready-for-review,done,failed}/*.md`
in **`created_at` order** (per plan line 72). Where `created_at` is
absent or unparseable, falls back to the file's mtime so the script never
crashes on malformed manifests.

Idempotent: if `by_canonical[id]` is already set, the file is skipped.

## 11. `node_modules/` policy in worktree

We add `proper-lockfile` to `package.json` but **do not commit**
`node_modules/` (which is gitignored at the repo root). The operator
will `npm install` on wake. The test suite invokes the public
`proper-lockfile` API; if `node_modules/` isn't populated when CI runs
the tests, the test file will fail with a clear `Cannot find module`
error rather than silently passing.

(Alternative: dynamic-import `proper-lockfile` with a graceful fallback.
Rejected — fallback to a hand-rolled lock would silently violate C17.)
