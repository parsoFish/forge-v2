## D-04 — graphify NOT used to generate the live graph; deterministic walker instead

graphify's `update` command only handles code files. graphify's
`extract` command requires either `--semantic <path>` (a JSON authored
by the assistant skill) or a `--backend` provider with an API key.

The available auth is the Claude Code OAuth token under
`~/.claude/.credentials.json`, which graphify **rejects** with
`invalid x-api-key` when passed as `ANTHROPIC_API_KEY`. Operator is
asleep — cannot ask for an API key.

**Decision:** ship a deterministic graph builder
(`orchestrator/brain-graph.ts`) that extracts the same structural edges
graphify would (frontmatter `related_themes`, wikilinks, `## Sources`
citations) without needing an LLM, and emit nodes/edges in the
**graphify schema** (`GraphNode` + `GraphEdge` from
`node_modules/graphifyy/dist/index.d.ts`).

This means a future operator pass with an API key can swap in
`npx graphify update brain/ --backend anthropic --all` at the same
`brain/graph.json` path and richer semantic edges will overlay
seamlessly — same node schema, same consumers, only the `generator`
field changes.

The SKILL.md documents this escalation explicitly.

## D-06 — brain-query rewrite is structural-routing, not graph-only

The plan said "consult graph first, fall back to keyword scan". I
interpreted this as **routing by question shape**:

- Structural phrasings ("bridges X and Y", "neighbours of …", "two
  hops from …") → graph-first; the graph contributes node ids that
  the agent reads to answer.
- Content-bearing phrasings (everything else) → keyword scan still
  the load-bearing path; the graph contributes a **structural recall
  widener** — running `neighbours` on a top keyword hit surfaces
  themes the keyword scan would miss (e.g. corrective antipattern).

This is gentler than a strict "graph first or bust" reading and
preserves the 18 narrative bench questions while making the 3 new
structural questions answerable. Documented in the rewritten
`skills/brain-query/SKILL.md` §Process.
