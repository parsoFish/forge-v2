# S1.3 — `assertLocalRemoteSynced` at dev-loop close — decisions

Stage S1.3 of the 2026-05-20 refinement batch. Operator asleep, autonomous
execution — judgments recorded here.

## Mandate

Add a call to `assertLocalRemoteSynced` at the close of `runDeveloperLoop`
(`orchestrator/phases/developer-loop.ts`), so a local↔remote divergence
becomes a hard, classified failure at the dev-loop boundary.

## Decisions

### D1. Keep the existing `enforceDevLoopCloseInvariant` call in `cycle.ts`

`orchestrator/cycle.ts:384-417` already wraps `pushInitiativeBranch` +
`assertLocalRemoteSynced` in `enforceDevLoopCloseInvariant`, called at
`cycle.ts:112` immediately after `runDeveloperLoop` returns. Per the
operator's standing instruction (`feedback_destructive_instruction_preserve_intent`):
**not removed, not changed.** The new dev-loop-internal call is
**additional**, not a replacement.

Rationale for keeping both:

- The cycle.ts call also gates a final `commitDevLoopBoundary` push +
  emits the `cycle.dev-close-invariant-ok` event with full ref-hash
  metadata. That commit-boundary scope belongs to cycle.ts, not to
  the dev-loop phase.
- The new dev-loop-internal call gives the dev-loop phase its OWN
  scoped failure signal — if a per-WI push silently failed mid-loop
  (network blip, etc.), the dev-loop fails fast at its own close
  with `phase: 'developer-loop'`, rather than bubbling out of an
  orchestrator-level assertion that the developer-loop phase passed.
  Two assertions, two phase-scoped events, no behavioural conflict
  (both invocations are idempotent reads against the same git state).

### D2. Event naming: `dev-loop.branch-divergence`

Forge's convention for dev-loop events is `dev-loop.<descriptor>`
(see `phases/developer-loop.ts:268`: `dev-loop.branch-pushed` /
`dev-loop.branch-push-failed`). The new event is
`dev-loop.branch-divergence` — matches that style, surfaces the
specific failure mode, easy to classify.

`event_type: 'error'` — divergence IS an error per the failure
classifier conventions.

Metadata mirrors the cycle-level success event's shape so post-mortems
can read the same ref hashes from either layer:

```
{
  branch, local_head, origin_head, main_head, merge_base, detail
}
```

### D3. Re-throw after emit — caller decides

Per the mandate. `runDeveloperLoop` already throws on total dev-loop
failure (line 302-306); a thrown invariant violation propagates up to
`cycle.ts:try/catch`, which calls `emitFailureClassification` —
classifier picks it up via `event_type: 'error'` in `developer-loop`
phase. (The classifier's `dev-loop-total-failure` matcher keys off
`phase === 'orchestrator'`, so a developer-loop-phase error doesn't
trigger that bucket. It currently falls into `unknown` / `agent-threw`.
Adding a new `FailureMode` is OUT OF SCOPE per the "small targeted bug
fix" mandate — the JSONL event is the classified signal; the
classifier table can grow in a follow-up if needed.)

### D4. Test approach: real tmp git repo

The mandate suggests a test-injection seam if available. `pr.ts:
assertLocalRemoteSynced` has none — it shells `git` directly via
`execFileSync`. Real tmp repos are already the pattern in
`pr.test.ts` and `closure.test.ts` (which uses identical
`makeRepoWithOrigin` shape). Re-use that shape.

### D5. Test file: new file `developer-loop-close-sync.test.ts`

There is no existing `developer-loop.test.ts`. A new dedicated file
is cleaner than a multi-concern bundle. Two tests: one for divergence
(throws + emits classified event), one for no-divergence (silent).

The new tests exercise only the close-step (extract the close
invariant into a tiny exported helper would be over-engineering for
this PR — instead the test exercises the public `runDeveloperLoop`
with a synthetic worktree containing one trivially-passing WI, and
asserts the close behaviour). To keep the test fast and SDK-free,
the test must avoid any real Ralph invocation. The simplest path:
test the close logic directly by extracting it into a private
helper (`assertDevLoopCloseSync`) and exercising the helper. That
helper is tiny — the public `runDeveloperLoop` already calls it.

**Chosen approach:** factor out a tiny `assertDevLoopCloseSync(worktreePath, logger, initiativeId, parentEventId)`
helper inside `developer-loop.ts`, call it at close, and unit-test
the helper with real tmp repos. The helper is a pure orchestration
of `assertLocalRemoteSynced` + an event emit + re-throw — easy to
test, no SDK, no Ralph.
